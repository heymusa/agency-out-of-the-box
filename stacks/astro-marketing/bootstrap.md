# Astro Marketing Stack — Bootstrap Procedure

> **Why this file exists.** Pinning framework versions inside this repo means the toolkit rots every 60–90 days. Instead, we run the **official `npm create astro@latest`** at init time and apply our overlay on top. This way the user always gets the current Astro version, and our overlay only carries what's actually ours (Tailwind tokens bridge, Layout, scaffold pages, design-system block wiring).
>
> The `design-builder` agent runs this procedure when `/build-agency-site` or `/new-client-site` first scaffolds an Astro project. Operators can also run it by hand.

## Step 1 — scaffold a fresh Astro project

Run the **current official Astro init** (don't pin versions in this repo — let `create-astro` pick the latest stable):

```bash
npm create astro@latest -- \
  --template minimal \
  --typescript strict \
  --install \
  --no-git \
  ./output/agency-site
```

Flags explained:
- `--template minimal` — empty starter, we add our own pages.
- `--typescript strict` — matches the strict frontmatter in our blocks.
- `--install` — installs deps so the next step works immediately.
- `--no-git` — the parent toolkit repo manages git; we don't want a nested one.

The agent should **not** hardcode an Astro major version. If `create-astro` defaults change (e.g., new Astro 6 prompt set), update this file's flags rather than vendoring a `package.json`.

## Step 2 — add the official integrations

Use Astro's first-party `astro add` command — it patches `astro.config.*` and `package.json` for you, with the correct current versions:

```bash
cd ./output/agency-site
npx astro add tailwind --yes
npx astro add mdx --yes
npx astro add sitemap --yes
```

If a future Astro release ships Tailwind 4 / Vite-plugin-tailwind by default, prefer that path (`astro add tailwind` continues to be the right entry point — it picks the supported integration automatically).

## Step 3 — apply the overlay

Copy our overlay files on top of the freshly-scaffolded project. Files in `overlay/` are intentionally **opinionated patches**, not full configs:

```bash
# from the toolkit repo root
cp -R stacks/astro-marketing/overlay/. ./output/agency-site/

# Symlink (or copy) the shared design-system so blocks resolve via @design-system/*
ln -s ../../design-system ./output/agency-site/design-system
```

Overlay contents:

| Overlay file | Purpose |
|---|---|
| `astro.config.mjs` | Our integration list + prefetch + static output. Replaces the boilerplate `astro add` produced. |
| `tailwind.config.ts` | Token bridge — imports `design-system/tokens.json`, switches palette by `ACTIVE_BRAND_HUE`. |
| `src/layouts/Layout.astro` | Shared layout: OG tags, Plausible, font preconnect, `cta_clicked` event delegation. |
| `src/pages/index.astro` | Reference homepage scaffold composed from `design-system/blocks/*` — every slot marked `<!-- TODO: ... DO NOT SHIP -->` so QA blocks deploy until real content lands. |
| `public/robots.txt`, `public/favicon.svg`, `public/images/hero-placeholder.svg` | Defaults the operator replaces. |

## Step 4 — wire the design-system path alias

Append to the freshly-bootstrapped `tsconfig.json`:

```jsonc
{
  "extends": "astro/tsconfigs/strict",
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@design-system/*": ["./design-system/*"],
      "@/*": ["./src/*"]
    }
  }
}
```

The `design-builder` agent does this with a JSON-merge, not a file replacement, so it survives upstream Astro changes to the default `tsconfig`.

## Step 5 — verify

```bash
cd ./output/agency-site
npm run dev      # http://localhost:4321 — should render the scaffold
npm run build    # type-check + production build
```

If `astro check` fails on the overlay, the most likely cause is a tokens-shape mismatch. Run `node -e "console.log(Object.keys(require('./design-system/tokens.json').color.brand))"` and confirm the active hue exists.

## Step 6 — pre-`/ship-it` checklist

- [ ] `agency-context.md` §13 has real domain + analytics ID.
- [ ] `astro.config.mjs` `site:` matches the real domain.
- [ ] `Layout.astro` `data-domain` attribute matches Plausible config.
- [ ] `public/robots.txt` `Sitemap:` URL matches real domain.
- [ ] No file contains `<!-- TODO: ... DO NOT SHIP -->` markers (the QA gate enforces this).

## What this overlay does NOT bring

- **No Astro version pin.** We trust `create-astro` to pick the right one.
- **No `package.json`.** Generated fresh per project.
- **No `tsconfig.json`.** Generated by `create-astro`, then patched in Step 4.
- **No node_modules.** Always installed locally per project.

This separation is intentional. The overlay stays small (under 10 files) and ages slowly. Framework boilerplate stays upstream where it belongs.
